# `.claude/handoffs/`

Context-transfer prompts generated by the `/handoff` skill. Each one lets a fresh Claude Code (or Cursor) session cold-start exactly where the previous session left off.

**The individual handoff files are gitignored.** Only this README and the symlink `latest.md` are tracked. The reason: each handoff captures a session's working state at a specific moment — staged files, open questions, branch SHAs — and that state is per-contributor, per-machine. A teammate's handoff isn't useful to another teammate, and committing one would pollute git history.

## Directory layout

```
.claude/handoffs/
├── README.md                    Tracked — this file
├── latest.md                    Tracked symlink → the most recent handoff
├── 2026-04-21-1432.md           Gitignored — actual handoff content
├── 2026-04-21-1805.md
└── ...
```

## File naming

`YYYY-MM-DD-HHMM.md` in local time. Claude Code usually runs in a single contributor's timezone per machine, so local time is the unambiguous reading; if CI ever generates a handoff it should use UTC and append `Z`.

## The symlink

`latest.md` always points at the most recent committed handoff. `/handoff` updates the symlink after writing a new entry. Since the symlink _target path_ is what's committed (not the target content), the symlink is safe to track even though its target is gitignored.

On a fresh clone, `latest.md` will point at a filename that doesn't exist locally (the previous contributor's file). That's fine — a new `/handoff` invocation rewrites the symlink on first use. `/phase-start` dispatches `/handoff` as part of its procedure to ensure a valid `latest.md` exists before real work begins.

## What a handoff contains

Per `skills/handoff/SKILL.md`, each file includes:

1. Reading list (files the next session must read, in order)
2. State summary (branch, last commit SHA, staged/uncommitted counts, active TodoWrite list)
3. What was done in this session
4. What's next — immediate step(s) per the active plan
5. Non-obvious decisions made this session (with ADR / learning pointers)
6. Open questions awaiting the maintainer
7. Verification recipe — concrete commands the new session runs to confirm it recovered cleanly
8. Paste-ready "continuation prompt" for a fresh chat

The "Paste this into the new chat" block is the critical deliverable. If the new session can paste it, read the three files it points at, run the verification recipe, and proceed — the handoff worked. If any step fails, the handoff is insufficient and the skill itself needs improvement.

## When handoffs get generated

- At natural stopping points (end of sub-phase; before closing a session you plan to resume later)
- When the `user-prompt-context-warn` hook flags context approaching the ~80% threshold
- As a dry-run at the end of each sub-phase to dogfood the pipeline (per ADR 0001 Phase 1 stop criterion)
- Explicitly via `/handoff` with an optional note argument

## Retention

No automatic pruning of local files. They accumulate on disk. Contributors may delete stale ones (`find .claude/handoffs -name '*.md' -mtime +30 -not -name README.md -not -name latest.md -delete`) — the `latest.md` symlink survives.

## Relationship to journal

`.claude/journal/` records _what happened_; `.claude/handoffs/` records _what you need to know to resume_. The journal is narrative and reflective; the handoff is procedural and forward-looking. Both are useful; neither replaces the other.

# Trading Domain

Application-specific rules for code that handles real-money trading. Expanded as the trading subsystems arrive in Phases 5–11. Placeholder-grade for now — the full set of invariants will materialize alongside the code.

This rule exists already (rather than being deferred) because writing it now forces decisions that would otherwise be made ad-hoc when trading code lands.

## Foundational invariants

These hold across every trading-adjacent module.

### 1. OANDA is the source of truth for positions and orders

- The database is a **cache** of OANDA state, not an authority
- Writes (place, close, modify) go to OANDA first; local state reflects the result only after OANDA confirms
- The daemon reconciles every 2 minutes via REST + in real time via the transaction stream
- No code constructs a "what is open" view from local state alone — always reconciled

### 2. `Trade.source` is always `"oanda"`; origin is in `metadata.placedVia`

- `source` identifies the trade repository (always OANDA)
- `metadata.placedVia` identifies who placed the trade: `"manual"`, `"tv_alert"`, `"trade_finder"`, `"trade_finder_auto"`, `"ai_trader"`, `"smart_flow"`
- Display labels and badge colors are derived from `placedVia` via `enrichSource()` (in `packages/db/src/trade-service.ts`, added in Phase 4)
- Metadata is **pre-seeded before the first reconcile** to avoid a race where a newly-placed trade syncs back as `manual` before its real origin is known

### 3. Pip precision is instrument-aware

- Never hardcode pip size
- Always route through `packages/shared/src/pip-utils.ts` (Phase 3)
- JPY-quote pairs (`*_JPY`): pip = 0.01
- Other majors and crosses: pip = 0.0001
- Metals, indices, and non-standard instruments have their own utilities — extend `pip-utils`, don't branch inline

### 4. Position sizing is risk-derived, deterministic, and shared

- Every subsystem that places an order uses `calculatePositionSize()` from `packages/shared/src/trading-core/risk-sizing.ts`
- No subsystem reinvents the formula
- Sizing is computed in code, **never delegated to an LLM** — LLM outputs are inputs to sizing (direction, SL, TP, confidence) but never the units themselves
- Fractional units are always floored, never rounded

### 5. Credentials flow only through the encrypted settings layer

- OANDA API keys, webhook tokens, Anthropic keys: stored in the `Settings` table, encrypted AES-256-GCM
- Reached via the DB service layer, never through env vars (see `11-env-vars.md`)
- Never logged. Never echoed in error responses. Never cached in plaintext for "performance"

### 6. Webhook tokens are per-source, per-user, and validated

- Generated by the app, never by the client
- Validated on receipt at both the CF Worker (fast reject) and the daemon (defense in depth)
- Rotated from the UI; old tokens invalidated immediately
- Appearance in logs, responses, or audit trails is a security violation

### 7. Every automated placement passes through a gate chain

In order, for every automated order:

1. **Settings gate** — is the strategy enabled? are we in the configured trading mode?
2. **Session gate** — is the market open? are we within the strategy's allowed session?
3. **Circuit breaker gate** — consecutive losses, daily drawdown, external pause
4. **Exposure gate** — concurrent trade cap, per-instrument cap, currency correlation cap
5. **Risk gate** — risk per trade, daily risk budget, spread-adjusted R:R
6. **News gate** — pending high-impact economic events within blackout window
7. **Instrument-specific gate** — JPY-pair cap, exotic-pair whitelist, weekend-closed markets
8. **Final validation** — SL is on the correct side of entry; TP is on the correct side; neither equals entry; units meet `minUnits`

Every gate is a separate function. Each returns a `GateResult` (`{ passed: boolean, reason?: string }`). Every rejection is logged with structured context. Every gate ships with unit tests.

### 8. Destructive actions have a grace period

Placing, modifying, or closing a trade within 60 seconds of the trade's creation is gated behind additional confirmation. This prevents a cascade of AI-driven "change your mind" actions on a newly-opened position.

### 9. Crash recovery is explicit

On startup, every subsystem with in-flight state (trade syncer, condition monitor, AI analysis pipeline) runs a recovery step:

- Reset stuck `executing` states older than N minutes to a retryable state
- Replay queued signals the CF Worker buffered during disconnect
- Re-seed metadata from audit trails where DB columns are null

No recovery step is optional. Agents ship them with the subsystem, not as an afterthought.

### 10. Every trade lifecycle event is broadcast

- Daemon emits structured WebSocket messages for: `trade_placed`, `trade_filled`, `trade_closed`, `trade_modified`, `order_cancelled`, `position_update`, `price_update`, `signal_received`, `circuit_breaker_tripped`, etc.
- Every event has a Zod schema in `packages/types`
- Every event's payload is logged (with redaction) to the audit trail
- Every event type has a contract test verifying the server's actual payload matches the declared schema

- **SmartFlow autonomous management** (Phase 11) — 4 scan modes, 9-gate pre-entry filter chain, tick-level management engine (breakeven, trailing, partials), AI monitor for major-move reactions
- **Circuit breakers** (Phase 5 onwards) — shared primitive in trading-core; per-subsystem instances with independent state; clock-injectable for test
- **Correlation engine** — currency-exposure tracker; same-currency same-direction cap; configurable per strategy
- **Reconciliation and metadata repair** — orphan close fetches OANDA for exit details before marking `closed`; metadata backfill scans recent placement audit trails
- **Partial closes and pyramiding** — position-slice accounting; MFE/MAE watermarks flushed every 30s
- **Management log** — every automated action (breakeven move, partial close, trail step, AI decision) appends a structured log entry on the trade row
- **Spread-adjusted R:R** — quoted R:R accounts for expected spread at entry and exit; minimum effective R:R floor
- **News gate** — FRED / economic calendar integration; configurable impact threshold; tight blackout window

Each of these gets a requirement in `docs/requirements/trading/`, a rule section added here, and a suite of tests before it can be considered shipped.

## What the reviewer agents check (trading-specific)

- Every order-placement path flows through the gate chain
- Every gate returns `GateResult` with explicit `reason` on rejection
- Every automated placement logs the full gate trace
- No hardcoded pip sizes
- No bypass paths around the shared sizing primitive
- No direct OANDA writes outside the dedicated OANDA client
- No trade mutation without an accompanying audit log entry
- All WebSocket events are schema-validated before broadcast
- No credential in a log line
- Every new subsystem ships with a documented recovery step

## Unknown territory

Any trading-related change that does not fit a pattern above triggers the unknown-territory protocol from `00-foundation.md`: stop, ask the maintainer, record the answer, proceed.
